# In-House POS System Implementation Plan

## Overview
Add a dedicated POS interface for restaurant staff to manage dine-in and takeaway orders directly from the app, separate from the customer-facing delivery system.

## Current State Analysis
- **User Roles**: Already defined (ADMIN, STAFF, CUSTOMER) in Prisma schema
- **Orders**: Support `DINE_IN` and `TAKEAWAY` types with `tableNumber` field
- **Tables**: `Table` model exists with `number` and `capacity` fields
- **Products**: Fully defined with categories, pricing, availability
- **Payments**: Mobile money (MTN_MOMO, AIRTEL_MONEY) and CASH support

## Proposed Solution

### 1. New `/pos` Route & Interface
A standalone POS page optimized for staff tablets/touchscreens:
- **Quick Order Creation**: Grid layout of products with large buttons
- **Table Selection**: For dine-in orders
- **Order Queue**: Real-time display of pending orders
- **Kitchen Tickets**: Print/view functionality
- **Payment Processing**: Cash, mobile money, split bills

### 2. Database Enhancements
```sql
-- Add POS-specific fields
ALTER TABLE "Order" ADD COLUMN "staffId" String?;  -- Who created the order
ALTER TABLE "Order" ADD COLUMN "printedAt" DateTime?;  -- Kitchen ticket timestamp
ALTER TABLE "Order" ADD COLUMN "servedAt" DateTime?;  -- Served to customer

-- Add staff notes for kitchen
ALTER TABLE "OrderItem" ADD COLUMN "staffNotes" String?;
```

### 3. API Endpoints to Add
- `POST /api/pos/orders` - Create order for dine-in/takeaway (no userId required for walk-in customers)
- `GET /api/pos/orders?status=PENDING` - Get orders for kitchen display
- `PUT /api/pos/orders/[id]/status` - Update order status (PREPARING -> READY)
- `GET /api/pos/tables` - Get table status (available/occupied)
- `POST /api/pos/print/[id]` - Generate print-friendly kitchen ticket

### 5. Implementation Steps

#### Phase 1: Core POS Page (Hours 1-2)
1. Create `/src/app/pos/page.tsx` with basic layout
2. Add POS-specific authentication guard (STAFF/ADMIN only)
3. Build product grid with quick-add functionality
4. Implement order cart with quantity controls

#### Phase 2: Order Management (Hours 2-3)
1. Create `/api/pos/orders/route.ts` for order creation
2. Add table selection dropdown for dine-in orders
3. Implement order submission with validation
4. Add loading states and error handling

#### Phase 3: Kitchen Integration (Hours 3-4)
1. Create kitchen display view at `/pos/kitchen`
2. Add WebSocket or polling for real-time updates
3. Implement status update buttons (PREPARING -> READY)
4. Add print functionality for kitchen tickets

#### Phase 4: Payments & Closing (Hours 4-5)
1. Add payment method selection (Cash/Mobile Money)
2. Implement split bill functionality  
3. Add order completion flow with printed receipt option
4. Provide both digital kitchen display AND printable kitchen tickets

## Key Questions for Clarification

1. **Table Management**: No reservation needed - just track tables via order numbers ✓

2. **Payment Flow**: Staff processes payments through POS (Cash/Mobile Money) ✓

3. **Kitchen Workflow**: Both digital display AND paper ticket printing ✓

4. **Staff Accounts**: Open-ended - any number of staff can have accounts

## Technical Considerations

- **Mobile-First**: POS should work on tablets (7-10 inch)
- **Offline Support**: Optional - could cache products locally
- **Receipts**: Generate receipts via browser print or PDF download
- **Performance**: Optimize for quick order entry (large touch targets)
